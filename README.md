# ci-cd-labtest ci
Порівняння GitHub-hosted та Self-hosted runner
Швидкість
Для порівняння швидкості було виконано кілька запусків CI pipeline на різних типах runner.
Pipeline, що запускався на GitHub-hosted runner, виконувався приблизно за 14–16 секунд.
Той самий pipeline, запущений на self-hosted runner на локальному комп’ютері, виконувався приблизно за 21 секунду.
У даному випадку GitHub-hosted runner показав трохи кращу швидкість виконання. Це пояснюється тим, що GitHub використовує оптимізовані сервери з достатньо потужними ресурсами. Швидкість self-hosted runner залежить від продуктивності локального комп’ютера та поточного навантаження системи.
Доступ до локальних ресурсів
GitHub-hosted runner працює у хмарному ізольованому середовищі, тому не має доступу до локальних файлів користувача або великих датасетів, що знаходяться на локальному диску.
Self-hosted runner запускається безпосередньо на локальному комп’ютері або сервері, тому має повний доступ до локальних ресурсів. Це дозволяє працювати з великими наборами даних, локальними файлами, базами даних або іншими сервісами, які вже розгорнуті на комп’ютері. Завдяки цьому self-hosted runner є більш гнучким для задач, пов’язаних із обробкою великих обсягів даних.
Ризики
GitHub-hosted runner є більш безпечним варіантом, оскільки він працює у тимчасовому ізольованому середовищі, яке автоматично очищається після завершення виконання job. Це зменшує ризик впливу виконуваного коду на систему.
Self-hosted runner має певні ризики. Оскільки workflow виконується на локальній машині, помилки в коді або небезпечні команди можуть вплинути на систему. Крім того, self-hosted runner може бути недоступним, якщо комп’ютер вимкнений, втрачено інтернет-з’єднання або процес runner не запущений (стан runner offline). Також користувач самостійно відповідає за оновлення програмного середовища та підтримку стабільності роботи runner.
